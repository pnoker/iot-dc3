# 规则告警链路优化

> 状态: 待补充设计。本文用于承接历史索引中已经出现的规则告警优化主题，后续应基于当前规则实现、缓存策略和通知链路补充完整方案。

## 背景

规则告警链路通常涉及点位值接收、规则匹配、状态转换、告警落库和通知投递。链路中任何一处缺少租户上下文、幂等控制或恢复判断，都可能导致误报、重复通知或跨租户数据风险。

本设计主题用于梳理规则告警从数据进入到状态恢复的完整路径，并为后续代码改造提供验收标准。

## 设计目标

- 规则匹配必须携带租户、设备、位号和规则上下文。
- 规则缓存需要有明确刷新策略和失效边界。
- 告警触发、持续、恢复和关闭语义必须可追踪。
- 通知投递应异步化，并保留失败重试和状态记录。
- 批量判断不能破坏单条告警的状态一致性。

## 需要确认的问题

| 问题 | 说明 |
|------|------|
| 规则缓存边界 | 以租户、设备、位号还是规则组为缓存 key |
| 窗口规则 | 滑动窗口、连续 N 次、持续时长等规则如何表达 |
| 恢复条件 | 恢复是否需要反向条件，是否要求持续正常一段时间 |
| 批量判断 | 批量点位值进入时如何保证规则结果和告警状态顺序 |
| 通知幂等 | 同一告警是否只通知一次，恢复是否单独通知 |
| 数据一致性 | 规则判断、告警落库和通知投递失败时如何补偿 |
