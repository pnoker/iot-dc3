# 实体告警统一表设计

> 状态: 待补充设计。本文用于承接历史索引中已经出现的 `dc3_entity_alarm` 主题，避免方案链路断裂。落地前需要结合当前数据表、事件模型和规则链路重新校准。

## 背景

IoT DC3 存在驱动、设备、位号和规则等多类事件来源。告警链路如果分散在多张表或多套状态模型中，会带来查询口径不一致、恢复逻辑重复、通知状态难追踪等问题。

实体告警统一表的目标，是为 point、device、driver 等实体提供统一的告警记录模型，并与状态表、规则判断、通知链路形成清晰边界。

## 设计目标

- 统一实体告警记录的基础字段、状态字段和租户边界。
- 区分当前状态事实与需要追踪的告警事件。
- 支持告警产生、确认、恢复、关闭和通知状态追踪。
- 兼容设备/驱动离线告警、故障告警、规则告警和自定义事件告警。
- 为后续 API、Web UI 和 Agentic Center 查询提供稳定模型。

## 需要确认的问题

| 问题 | 说明 |
|------|------|
| 实体类型枚举 | point、device、driver 是否足够，是否需要 profile、tenant 或 gateway |
| 告警状态机 | pending、active、acknowledged、recovered、closed 等状态如何转换 |
| 恢复语义 | 恢复是新事件、原记录状态更新，还是两者都保留 |
| 去重窗口 | 同一实体同一规则重复触发时如何合并或抑制 |
| 通知状态 | 通知记录是否进入统一表，还是使用独立通知历史表 |
| 租户索引 | 查询和写入必须保留租户维度，索引需要覆盖高频查询 |
