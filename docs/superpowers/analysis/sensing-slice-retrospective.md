## 2. 回答 design 第 8 节复盘标准（逐条）

### Q1：verify 是否拦住幻觉？

**答：拦住；本轮零拦截事件，机制有效。**

Task 1 主 agent 用 `pdftotext -layout` 抽 VOL1 p124、CPS p190、CPS p275 三个 cited 页，逐字 grep 原句片段**全命中**（含 CPS 异体字规范化后的比对）。无伪造、无错页、无返工。这与协议层"零幻觉流入产出"一致——**命令行 pdftotext 独立核对**这一机制在本切片复用并再次验证有效。implementer 的 1 条诚实 N/A（奈奎斯特/混叠）从源头消除了最常见的幻觉来源。

### Q2：三分类是否真能区分真缺陷 vs 工程取舍？

**答：能区分；本轮的最大价值恰恰是"不硬凑真缺陷"得到正确兑现。**

本轮真缺陷 = **0**。最有诱惑力被判真缺陷的两处，均经论证降级：

- **D-2（浮点 round 绕行 primitive）**：`BigDecimal→double→String→BigDecimal` 看似精度有损，但实跑 `RoundTest` 覆盖 17 位有效数字、`2.675` tie、float 7 位精度边界后，结果与直接 `setScale(HALF_UP)` **一致**——故降级为技术债（代码意图与路径不一致），而非真缺陷。判据：**实跑验证，不靠直觉判精度问题**。
- **D-1（byte 截断）**：`(byte) dto.getValueDecimal()` 静默环绕看似真缺陷，但发生在 gRPC facade 边界而非标定核心，且实际 `valueDecimal` 范围远小于 `[-128,127]`，归为技术债（隐患债）而非当前真缺陷。

**有意简化 vs 真缺陷的边界**：S-1（仅线性标定 `y=mx+b`）——线性只是"输出-输入关系"的一种形式（条目 4.1），不支持多项式/查表/分段属工程取舍而非缺陷，因大部分工业位号线性足够。这与协议层 S1/S2（驱动 `throw NotImplemented` 快速失败）的判据同构：**如实记录设计边界，不判错**。

**结论**：三分类有效。本轮证明的关键点是——**0 真缺陷是合法结论，三分类的价值正在于允许"诚实评估、不硬凑"**（见第 4 节）。

### Q3：知识条目表格式是否可复用？

**答：完全可复用，字段零改动。**

`概念 | 可引用原句 | 书名/作者/出版社·年份/章节/页码 | 对应 docs 锚点 | verify状态` 五字段在协议层定义后，本轮**直接复用未改一字**。跨书源（VOL1 / CPS）、跨主题（分类/信号链/A/D/标定）、跨页码体系（VOL1 双标印刷+PDF / CPS 仅 PDF）均稳定承载。N/A 条目沿用协议层 `N/A | <原因> | ...` 写法（条目 3.3）。

协议层复盘提到的两个 Minor 改进点（N/A 格式标准化 / 来源定位子表），本轮 N/A 仅 1 条未触发格式压力；来源定位子表本轮简化为"章节→PDF 页"单表（书源仅 2 本，无需协议层那种"书源×章节×页码×相关性"矩阵）。**结论：格式可复用，按书源数量自适应选用单表或矩阵**。

### Q4：单页成本估算铺开是否可接受？

**答：可接受，且显著低于协议层上限。**

单页约 3 子 agent / 20–40k token，约为协议层单页成本的 1/3（协议层 10–12 子 agent / 30–50k token）。差异主因：

- **单页单主题 vs 多页多主题**：本轮 1 页对应 4 个紧密相关子主题、2 本对口书，书源定位一次完成；协议层 3 页对应 fieldbus+iot-protocols+drivers 多个不相邻主题、4 本通论书。
- **无独立 verify 子 agent 编排**：本轮 verify 由主 agent 内联 pdftotext 抽 3 页承担（cited 页少），未起独立 verify 子 agent；协议层 defects 19 处 `文件:行` 全核需独立 verify 子 agent。
- **缺陷分析复用既有脉络**：标定换算链路集中在 `TypedValueConverter`，源码体量小、无跨模块追溯。

**外推**：若其余 foundations 页单页成本与本轮相当，逐页铺开的子 agent 调用量约为协议层外推值的 1/3。但仍建议按"书源 → 主题群"批量推进（见第 5 节），以摊薄书源定位的固定成本。

## 3. 与协议层切片对比

### 3.1 书源驱动策略的有效性（vol1/vol2 重叠弃用经验再次命中）

协议层复盘指出"同 ISBN 的 vol1/vol2 PDF 常为同一本书的编码差异重复，取一份即可"。本轮 **VOL1/VOL2 再次命中同一规律**：经 p150 正文逐字比对，确认 VOL2 是 VOL1 逐字重复（同 ISBN/主编/出版社/年份，md5 不同仅为 PDF 编码差异），**弃用 VOL2 避免重复条目**。

这条经验已两次验证（协议层 + sensing），可固化为流程：**遇到同系列 vol1/vol2 PDF，先抽 1 页正文逐字比对，确认重复即弃用一本**，避免重复建条目浪费工作量。

### 3.2 目标②薄印证（4 真缺陷 vs 0 真缺陷）

协议层目标②发现 **4 真缺陷**（MQTT health 桩伪造 online / 资源未关闭 / 错误不退避 等），sensing 目标②发现 **0 真缺陷**。这一反差是**诚实且正确**的：

- **协议层靶子是连接管理**：OPC-UA 连接风暴、MQTT broker 不可达仍自报 online——分布式资源生命周期，缺陷密度天然高。
- **sensing 靶子是标定换算**：`工程值 = 原始值 × multiple + baseValue` 是简单线性变换，实现质量好——`BigDecimal` 全程精确算术、null 标定参数有 `Optional.orElse(DEFAULT_*)` 兜底、`HALF_UP` rounding 无系统性 DOWN 截断偏差、类型转换显式抛 `TypeException/OutRangeException` 不静默吞异常。

design 第 3 节预警过"目标②可能比协议层薄"，本轮实证兑现。**0 真缺陷不是漏看**：D-2（浮点 round 绕行）是最可疑的精度问题，已写 `/tmp/RoundTest` 实跑对比，确认当前不产生算错才降级。这是"实跑验证"而非"扫一眼放过"。

**结论**：薄印证成立。代码缺陷密度取决于代码本身的复杂度与资源管理面，不取决于切片方法论——**方法论的作用是如实暴露，而非保证每个切片都有料**。

## 4. 关键发现：0 真缺陷是三分类的合法产出

本轮最重要的发现不是某条具体缺陷，而是——**不是所有代码都有真缺陷，三分类的价值正在于允许这一结论合法存在**。

若只有"真缺陷 / 非真缺陷"二分，或带 KPI 式"每个切片必须产出 N 条真缺陷"的隐性压力，会有两种坏结果：
1. **书生气误判**：把 D-2（绕行 round）或 D-1（byte 截断）硬凑成真缺陷，写出"精度损失严重"这类经不起实跑的结论。
2. **为了凑数灌水**：把 S-1（仅线性标定）这种工程取舍判成缺陷，污染缺陷清单的可信度。

三分类通过设立【有意简化】和【技术债】两个出口，让 implementer 有地方如实安放"知道欠账但当前不算错"的条目，从而守住【真缺陷】的含金量。**协议层 4 真缺陷与 sensing 0 真缺陷都是诚实结论**——后者证明这套方法不靠堆数量取信。

## 5. 是否继续铺开 + 下一批建议

### 5.1 结论：继续铺开，按"书源 → 主题群"批量推进

四个复盘标准的答案：verify 零幻觉（有效）、三分类可区分且允许 0 真缺陷（判据明确）、条目表可复用、单页成本显著低于协议层。**方法论再次验证通过，可铺开**。

### 5.2 下一批主题建议（结合书源覆盖度）

本轮 2 本书源（VOL1 / CPS）覆盖的剩余主题，以及协议层已验证书源的延伸：

1. **MEMS / 传感器工艺**（CPS 第6章有 MEMS 内容，sensing.md 已有 MEMS 段未注入）——同书源边际成本低，可顺手补。
2. **动态特性 / 频率响应**（VOL1 5.1.2 动态特性已读但本轮未建条目）——同书源延伸。
3. **MQTT / CoAP / LPWAN 深化**（协议层复盘建议的优先项，书源最充足）——跨切片衔接。

**暂缓或需另配书源的主题**：
- 奈奎斯特定理 / 混叠（条目 3.3 N/A）——VOL1/CPS 均无清晰定义，需信号与系统或数据采集教材。
- 非线性标定（热电偶/RTD 分度表、S-1 提及）——需温度传感器专著或计量手册。
